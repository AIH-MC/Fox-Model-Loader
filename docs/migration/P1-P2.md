# P1-P2 NeoForge Migration Notes

Completed:

- Copied the common Java and resource trees into the standalone NeoForge project.
- Removed direct Fabric API, Fabric Loader, Fabric entrypoint, Fabric mixin config, and Cardinal Components references from `src/main`.
- Kept local `dev.architectury` compatibility shims only as internal migration bridges; they no longer call Fabric APIs.
- Registered NeoForge config specs through `ModContainer#registerConfig`.
- Replaced player/entity capability access with NeoForge Data Attachments.
- Added a NeoForge payload bridge for the existing YSM discriminator-based packet registry.
- Routed server-to-client, client-to-server, all-player, tracking-entity, and tracking-entity-and-self sends through NeoForge packet distributors.
- Bridged NeoForge server events back to the existing common event handlers:
  - server lifecycle
  - command registration
  - player join, quit, and clone
  - entity join level
  - server post tick
  - server reload listeners
- Bridged NeoForge client events back to existing client handlers:
  - client lifecycle
  - client tick
  - client commands
  - client player login, logout, and clone
  - key mapping registration
  - client reload listeners
- Relaxed network connection checks so they no longer depend on disabled accessor mixins.

Verified:

- `rg` found no `net.fabricmc`, `.fabric.`, `org.ladysnake`, `fabric.mod.json`, `FabricLoader`, or Forge Config API Port references under `src/main`.
- `gradlew compileJava --no-configuration-cache` succeeds.
- `gradlew build --no-configuration-cache` succeeds.

Deferred:

- Data Attachment persistence over logout/rejoin still needs a dedicated serializer or save/load event path.
- Third-party compatibility modules are currently no-op fallbacks.
- WebP and AVIF decoder integration is stubbed.
- Rendering/input mixins remain disabled for P3.
