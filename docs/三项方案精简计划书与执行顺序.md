# 三项方案精简计划书与执行顺序

> 项目：`D:\OYSM\openysm-26.1.2`  
> 结论日期：2026-05-31  
> 范围：资源站功能、性能与多模型载入优化、NeoForge 独立迁移  
> 约束：本文是计划书，不包含代码修改。

## 一、总体判断

三项方案都可行，但不应同时推进。

推荐执行顺序：

```text
1. 性能与多模型载入内存优化
2. 资源站功能
3. NeoForge 独立迁移
```

理由：

```text
性能优化先做：
    当前模型加载、纹理、GPU/direct buffer 生命周期会直接影响稳定性。
    资源站会引入更多模型下载和导入，如果先不压低内存峰值，资源站更容易放大问题。

资源站第二做：
    当前项目已有 DownloadScreen、ModelUploadScreen、ModelUploadSession、ServerModelManager 等基础。
    在性能底座稳定后，资源站可以复用现有导入/上传/重载/同步链路。

NeoForge 迁移最后做：
    它是独立项目级迁移，不是小功能。
    会重写构建、配置、事件、网络、数据附件、Mixin 入口。
    如果先做，会让当前 Fabric 版功能开发和问题定位变慢。
```

如果目标是尽快给玩家看到新功能，可以把“资源站 P0”提前到性能 P0 之后执行，但不建议在没有完成性能 P0 前直接上线资源站。

## 二、计划书一：性能与多模型载入内存优化

### 目标

降低大量模型同时载入时的 Java Heap、Direct Memory、GPU 资源峰值，减少长期常驻内存和渲染热路径开销，同时保留 GPU Renderer、Native Renderer、Java fallback。

### 当前依据

当前项目已有：

```text
ClientModelManager 单线程模型解析池
OuterFileTexture 持有纹理 byte[]
GpuRenderPath / GpuMesh GPU 渲染路径
GpuMesh.dispose() 可释放 GL buffer、VAO、native pointer、perFrameBoneBuffer
ModelUploadSession / ServerModelManager 导入链路
```

关键风险点：

```text
Files.readAllBytes + YsmCrypt.read + parse 可能同时持有多份大 byte[]
OuterFileTexture.data 是 final byte[]，上传 GPU 后仍长期占用 heap
GpuMesh 每模型持有 perFrameBoneBuffer direct buffer
modelAssemblyMap 长期持有所有完整 ModelAssembly
GPU 状态绑定每次渲染重复执行
```

### P0：先做稳定性和可观测

1. 增加 `ModelMemoryProfiler`。
2. 日志覆盖读取、解密/解压、解析、纹理解码、纹理上传、GPU Mesh 构建、加入缓存、卸载。
3. 配置默认关闭。
4. 缩短 `fileBytes`、`decompressed`、导入 `data` 的生命周期。
5. 避免 lambda / future 长时间捕获大 byte[]。
6. `OuterFileTexture` 上传成功后，在配置开启时释放 Java byte[]。
7. 确认模型移除、退出世界、资源重载时会释放纹理、GpuMesh、direct buffer。

### P1：减少渲染和加载浪费

1. 对资源站下载、模型导入、解析、纹理上传做明确限流。
2. GPU 渲染路径增加帧级状态缓存，减少重复 `glUseProgram`、`bindTexture`、`bindSampler`、`bindSSBO`。
3. 纹理加载时生成 `AlphaProfile`，纯不透明模型跳过透明 pass。
4. 检查 Java fallback / Native fallback 热路径，复用 Matrix、Vector、数组等临时对象。

### P2：模型生命周期管理

1. Lazy Loading：进服只建立模型索引，附近玩家、当前玩家、GUI 预览再完整加载。
2. `ModelAssembly` / `Texture` / `GpuMesh` 加 LRU 卸载。
3. 远离玩家、长期不用、超过模型数量或内存预算时释放完整模型。
4. 本地玩家当前模型、正在渲染模型、GUI 正在预览模型不可淘汰。

### P3：高风险优化，仅评估后做

1. Shared `BoneUploadScratch` 或 `BoneRingBuffer`。
2. GPU Mesh 构建成功后，在低内存模式下释放 CPU baked mesh。
3. GeoModel compact 化。
4. Compute Skinning。

### 验收标准

```text
单模型加载、切换、资源重载正常
20/50/100 个模型加载时 heap 峰值下降
Direct Memory 不持续增长
退出世界后 GPU/direct 资源可释放
GPU Renderer / Native Renderer / Java fallback 都可用
Iris / Shaderpack 不崩溃
```

### 不做

```text
不使用 System.gc() 作为核心方案
不删除 fallback
不在后台线程调用 OpenGL
不一次性重写模型格式
不先做多线程渲染
```

## 三、计划书二：资源站功能

### 目标

在模型选择界面增加“资源站”，支持浏览远程模型资源，下载 `.ysm` / `.zip`，并复用现有导入、上传、服务端重载、客户端同步链路。

### 当前依据

当前项目已有：

```text

DownloadScreen：已有 Coming Soon 占位页
ModelUploadScreen：已有导入 UI、进度、错误状态参考
ModelUploadSession：已有客户端上传会话

ClientModelManager.importLocalModel / reloadLocalModels：已有本地导入和扫描能力
GeneralConfig / ModConfigSpec：已有配置模式
```

### 核心原则

不要把“下载到客户端本地 custom 目录”作为主流程。

正确主流程：

```text
资源站下载 bytes
-> 本地校验/预加载
-> ModelUploadSession.start(modelId, fileName, bytes)
-> 服务端 finishModelUpload
-> 服务端写入 config/yes_steve_model/custom
-> 服务端 reload
-> 服务端同步给客户端
```

原因：

```text
单机：客户端目录和服务端目录通常是同一环境，可以生效。
远程服务器：客户端本地目录不是服务器目录，服务端扫描不到。
```

## 四、计划书三：NeoForge 独立迁移

### 目标

将 Fox Model Loader 迁移为独立 NeoForge 项目，而不是在当前 Fabric 项目中混合 Fabric / NeoForge。

```text

root build.gradle 对 subprojects 统一应用 fabric-loom

common 内存在 Fabric / Architectury shim
Platform 直接依赖 FabricLoader
fabric.mod.json、Fabric entrypoint、Fabric mixin 配置都在 fabric 子项目
```

因此不建议在当前工程内新增 `neoforge` 子项目。

### 推荐结构

```text

    保留 Fabric 版

Fox-Model-Loader-NeoForge/

```


## 六、动画流畅度优化补充记录（2026-06-01）

### 执行边界

本轮只执行低风险和可小步验证的动画流畅度优化内容。

明确不做以下高风险项：

```text
不做 AnimationPoseCache 大规模姿态缓存重构
不做 prev/curr/render boneParams 全链路插值重构
不做动画 clip 预采样/烘焙
不做新的动画 LOD 系统
不做武器动作状态隔离重构
不重写 GPU / Native / Java fallback 渲染路径
```

原因：

```text
当前代码中 partialTick 并非完全丢失，AnimationEvent 初始 currentTick 已包含 tickCount + frameTime。
后续 currentTick 会被替换为内部 seekTime，因此低帧感需要先通过日志确认是 seekTime 推进、刷新率限流、同帧重复计算还是 controller 采样导致。
在没有证据前直接做姿态缓存、烘焙或 LOD，容易扩大风险并破坏第一人称、第三人称、GUI 预览、GPU Renderer、Native Renderer、Java fallback 和 shaderpack 路径。
```

### 已完成内容

已完成 P0 诊断基线，不改变动画行为：

```text
新增 AnimationFrameProfiler
新增动画帧分析配置项，默认关闭
新增动画详细调试日志配置项，默认关闭
新增同帧重复动画计算警告配置项
将上述三个配置项加入设置界面的“实验性测试”分组
补充 en_us / zh_cn 本地化
日志统一使用 [YSM-ANIM] 前缀
```

相关配置项：

```text

ExperimentalTesting.AnimationDebugLog=false
ExperimentalTesting.WarnRepeatedAnimationEvaluation=true
```

设置界面位置：

```text

  AnimationFrameProfiler（动画帧分析）
  动画详细调试日志
  重复动画计算警告
```

### GitHub 中国大陆加速策略

```text
用户仍输入标准 https://github.com/... 地址。
只接受 github.com 仓库地址作为 GitHub 源，不解析 GitHub HTML。
GitHub API 默认优先尝试代理前缀，再回退官方 API。
GitHub raw 文件下载优先尝试 jsDelivr gh CDN。
jsDelivr 失败后尝试代理前缀。
代理失败后回退 raw.githubusercontent.com 原始地址。
默认代理前缀写入 resource_station.properties：
  https://gh.llkk.cc/
  https://gh-proxy.com/
  https://gh.ddlc.top/
可通过配置文件调整 preferGithubAccelerator 和 githubAccelerators。
```

### 安全与边界

```text
下载文件仍限制 .ysm / .zip。
下载体积仍受 maxDownloadBytes 限制。
modelId 仍通过安全归一化生成。
中文、日文、空格等文件名只用于显示名；无法生成安全 modelId 时回退 sha1。
UI 关闭后后台回调不再回写已关闭界面。
不做 GitHub 登录/token。
不做评分、评论、收藏。
不做断点续传。
不做文件夹模型递归打包下载。
不新增 Mixin/Coremod/ASM。
```
