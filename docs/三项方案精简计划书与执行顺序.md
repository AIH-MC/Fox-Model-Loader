# 三项方案精简计划书与执行顺序

> 项目：`D:\OYSM\openysm-26.1.2`  
> 结论日期：2026-05-31  
> 范围：资源站功能、性能与多模型载入优化、NeoForge 独立迁移  
> 约束：本文是计划书，不包含代码修改。

## 一、总体判断

三项方案都可行，但不应同时推进。

推荐执行顺序：

```text
1. 性能与多模型载入内存优化
2. 资源站功能
3. NeoForge 独立迁移
```

理由：

```text
性能优化先做：
    当前模型加载、纹理、GPU/direct buffer 生命周期会直接影响稳定性。
    资源站会引入更多模型下载和导入，如果先不压低内存峰值，资源站更容易放大问题。

资源站第二做：
    当前项目已有 DownloadScreen、ModelUploadScreen、ModelUploadSession、ServerModelManager 等基础。
    在性能底座稳定后，资源站可以复用现有导入/上传/重载/同步链路。

NeoForge 迁移最后做：
    它是独立项目级迁移，不是小功能。
    会重写构建、配置、事件、网络、数据附件、Mixin 入口。
    如果先做，会让当前 Fabric 版功能开发和问题定位变慢。
```

如果目标是尽快给玩家看到新功能，可以把“资源站 P0”提前到性能 P0 之后执行，但不建议在没有完成性能 P0 前直接上线资源站。

## 二、计划书一：性能与多模型载入内存优化

### 目标

降低大量模型同时载入时的 Java Heap、Direct Memory、GPU 资源峰值，减少长期常驻内存和渲染热路径开销，同时保留 GPU Renderer、Native Renderer、Java fallback。

### 当前依据

当前项目已有：

```text
ClientModelManager 单线程模型解析池
OuterFileTexture 持有纹理 byte[]
GpuRenderPath / GpuMesh GPU 渲染路径
GpuMesh.dispose() 可释放 GL buffer、VAO、native pointer、perFrameBoneBuffer
ModelUploadSession / ServerModelManager 导入链路
```

关键风险点：

```text
Files.readAllBytes + YsmCrypt.read + parse 可能同时持有多份大 byte[]
OuterFileTexture.data 是 final byte[]，上传 GPU 后仍长期占用 heap
GpuMesh 每模型持有 perFrameBoneBuffer direct buffer
modelAssemblyMap 长期持有所有完整 ModelAssembly
GPU 状态绑定每次渲染重复执行
```

### P0：先做稳定性和可观测

1. 增加 `ModelMemoryProfiler`。
2. 日志覆盖读取、解密/解压、解析、纹理解码、纹理上传、GPU Mesh 构建、加入缓存、卸载。
3. 配置默认关闭。
4. 缩短 `fileBytes`、`decompressed`、导入 `data` 的生命周期。
5. 避免 lambda / future 长时间捕获大 byte[]。
6. `OuterFileTexture` 上传成功后，在配置开启时释放 Java byte[]。
7. 确认模型移除、退出世界、资源重载时会释放纹理、GpuMesh、direct buffer。

### P1：减少渲染和加载浪费

1. 对资源站下载、模型导入、解析、纹理上传做明确限流。
2. GPU 渲染路径增加帧级状态缓存，减少重复 `glUseProgram`、`bindTexture`、`bindSampler`、`bindSSBO`。
3. 纹理加载时生成 `AlphaProfile`，纯不透明模型跳过透明 pass。
4. 检查 Java fallback / Native fallback 热路径，复用 Matrix、Vector、数组等临时对象。

### P2：模型生命周期管理

1. Lazy Loading：进服只建立模型索引，附近玩家、当前玩家、GUI 预览再完整加载。
2. `ModelAssembly` / `Texture` / `GpuMesh` 加 LRU 卸载。
3. 远离玩家、长期不用、超过模型数量或内存预算时释放完整模型。
4. 本地玩家当前模型、正在渲染模型、GUI 正在预览模型不可淘汰。

### P3：高风险优化，仅评估后做

1. Shared `BoneUploadScratch` 或 `BoneRingBuffer`。
2. GPU Mesh 构建成功后，在低内存模式下释放 CPU baked mesh。
3. GeoModel compact 化。
4. Compute Skinning。

### 验收标准

```text
单模型加载、切换、资源重载正常
20/50/100 个模型加载时 heap 峰值下降
Direct Memory 不持续增长
退出世界后 GPU/direct 资源可释放
GPU Renderer / Native Renderer / Java fallback 都可用
Iris / Shaderpack 不崩溃
```

### 不做

```text
不使用 System.gc() 作为核心方案
不删除 fallback
不在后台线程调用 OpenGL
不一次性重写模型格式
不先做多线程渲染
```

## 三、计划书二：资源站功能

### 目标

在模型选择界面增加“资源站”，支持浏览远程模型资源，下载 `.ysm` / `.zip`，并复用现有导入、上传、服务端重载、客户端同步链路。

### 当前依据

当前项目已有：

```text
PlayerModelScreen：模型选择界面，可增加入口
DownloadScreen：已有 Coming Soon 占位页
ModelUploadScreen：已有导入 UI、进度、错误状态参考
ModelUploadSession：已有客户端上传会话
ServerModelManager.finishModelUpload：已有服务端校验、写入 CUSTOM、重载、同步
ClientModelManager.importLocalModel / reloadLocalModels：已有本地导入和扫描能力
GeneralConfig / ModConfigSpec：已有配置模式
```

### 核心原则

不要把“下载到客户端本地 custom 目录”作为主流程。

正确主流程：

```text
资源站下载 bytes
-> 本地校验/预加载
-> ModelUploadSession.start(modelId, fileName, bytes)
-> 服务端 finishModelUpload
-> 服务端写入 config/yes_steve_model/custom
-> 服务端 reload
-> 服务端同步给客户端
```

原因：

```text
单机：客户端目录和服务端目录通常是同一环境，可以生效。
远程服务器：客户端本地目录不是服务器目录，服务端扫描不到。
```

### P0：最小可用

1. 新增 `ResourceStationScreen`。
2. `DownloadScreen` 改为 `ResourceStationScreen` 的兼容入口。
3. `PlayerModelScreen` 增加资源站按钮。
4. 新增 `ResourceStationConfig`，保存默认 URL、当前选中 URL、超时、最大下载大小。
5. 新增 `ModelRepoEntry`。
6. 新增 `ModelRepoClient`。
7. 支持 OpenYSM `index.json`。
8. 支持 `.ysm` / `.zip` 下载。
9. 下载、列表刷新放后台线程。
10. UI 更新切回 Minecraft 主线程。
11. 下载完成后调用 `ModelUploadSession.start(...)`。
12. 界面监听 `ModelUploadSession.Listener` 显示上传状态。
13. 增加 `zh_cn` / `en_us` 语言键。

### P1：GitHub 兼容

支持：

```text
https://github.com/{owner}/{repo}
https://github.com/{owner}/{repo}/tree/{branch}
https://github.com/{owner}/{repo}/tree/{branch}/{path}
```

要求：

```text
只接受 https://github.com
使用 GitHub Contents API，不解析 HTML
仓库根地址优先读取默认分支，或第一版先尝试 main
递归列出 .ysm / .zip
处理 GitHub API 错误和速率限制
```

默认资源站：

```text
https://github.com/Elaina69/Yes-Steve-Model-Repo/tree/main
https://github.com/sdf123098/YSM-Model
```

### P2：体验完善

1. URL 添加、删除、切换。
2. 搜索、排序、过滤。
3. 文件大小和下载进度。
4. 预览图、作者、标签、描述。
5. 多任务下载队列。
6. 失败重试。

### 安全要求

```text
禁止路径穿越：../、..\
禁止绝对路径
modelId 只允许 [a-z0-9_./-]+
中文、日文、空格文件名只作为显示名
无法生成安全 modelId 时使用稳定 hash 兜底
限制最大下载体积
校验扩展名
关闭界面后旧后台任务不得回写已关闭界面
```

推荐兜底：

```text
显示名：绯英_0.4.ysm
modelId：repo/sdf123098/ysm-model/<sha1前12位>
上传文件名：<sha1前12位>.ysm
```

### 验收标准

```text
模型选择界面能打开资源站
资源站能返回模型选择界面
默认 URL 可见且可编辑
index.json 能列出 .ysm / .zip
GitHub 根地址和 tree URL 能列出 .ysm / .zip
下载不阻塞主线程
导入走 ModelUploadSession
远程服务器场景不会误写客户端本地 custom
成功后模型列表自动刷新
非法 URL、网络失败、下载失败、导入失败都有提示
```

### 不做

```text
不做 GitHub 登录/token
不做评分评论收藏
不做断点续传
不做文件夹模型递归打包下载
不新增 Mixin/Coremod/ASM
```

## 四、计划书三：NeoForge 独立迁移

### 目标

将 Fox Model Loader 迁移为独立 NeoForge 项目，而不是在当前 Fabric 项目中混合 Fabric / NeoForge。

### 当前依据

当前仓库是 Fabric-only：

```text
settings.gradle 只 include fabric
root build.gradle 对 subprojects 统一应用 fabric-loom
fabric/build.gradle 依赖 Fabric API、Cardinal Components API、Forge Config API Port
common 内存在 Fabric / Architectury shim
Platform 直接依赖 FabricLoader
fabric.mod.json、Fabric entrypoint、Fabric mixin 配置都在 fabric 子项目
```

因此不建议在当前工程内新增 `neoforge` 子项目。

### 推荐结构

```text
Fox-Model-Loader/
    保留 Fabric 版

Fox-Model-Loader-NeoForge/
    独立 NeoForge 版
```

NeoForge 版使用标准单加载器结构：

```text
src/main/java
src/main/resources
META-INF/neoforge.mods.toml
yes_steve_model.mixins.json
```

### P0：空 NeoForge 项目

1. 新建独立目录。
2. 使用 NeoForge MDK / ModDevGradle。
3. 配好 `settings.gradle`、`build.gradle`、`gradle.properties`。
4. 新增空 `@Mod` 主类。
5. 新增 `neoforge.mods.toml`。
6. 新增空 mixin json。
7. `runClient` 能进主菜单。

不要一开始复制全部源码。

### P1：复制 common 并清理依赖

1. 复制 `common/src/main/java` 到 `src/main/java`。
2. 复制 `common/src/main/resources` 到 `src/main/resources`。
3. 删除或替换：

```text
net.fabricmc.*
net.fabricmc.fabric.api.*
org.ladysnake.cca.*
dev.architectury.* shim
Forge Config API Port
Fabric entrypoint
fabric.mod.json
```

4. 编译失败按模块分类，不要一次性硬修所有错误。

### P2：平台能力替换

按顺序迁移：

```text
配置系统 -> NeoForge 原生 Config
数据组件 -> NeoForge Data Attachments
网络 -> NeoForge Payload Networking
事件 -> NeoForge Event Bus
命令 -> NeoForge command registration
资源重载 -> NeoForge reload listener
```

先跑通最小闭环：

```text
客户端选择模型
-> 服务端收到
-> 服务端保存/广播
-> 其他客户端显示变化
```

### P3：渲染与 Mixin

逐个启用：

```text
玩家第三人称渲染
第一人称手部渲染
键盘输入
鼠标输入
HUD
声音
实体效果
```

每启用一个 mixin，都运行客户端和服务端测试。

### 高风险点

```text
Cardinal Components API -> Data Attachments
玩家死亡/克隆/重进后的模型数据保留
Payload 注册和同步时机
专用服务器误加载 client-only 类
Mixin 目标签名变化
渲染线程和 OpenGL 调用位置
zstd-jni / ImageStream 打包
```

### 验收标准

```text
runClient 可进入主菜单
runServer 可启动专用服务器
配置文件正常生成
模型选择 UI 可打开
客户端 A 换模型，客户端 B 可见
玩家退出重进后模型保留
死亡/重生后模型数据保留
第三人称和第一人称渲染正常
专用服务器无 MinecraftClient 类加载错误
```

### 不做

```text
不在原 Fabric 工程里混合 NeoForge
不继续保留 Fabric API / CCA / Forge Config API Port
不试图同时兼容 Fabric 和 NeoForge
不一次性启用所有 Mixin
```

## 五、最终执行顺序

### 推荐路线

```text
第一阶段：性能 P0
    ModelMemoryProfiler
    byte[] 生命周期缩短
    OuterFileTexture 上传后释放 byte[]
    卸载/退出世界资源释放检查

第二阶段：资源站 P0
    ResourceStationScreen
    index.json
    .ysm/.zip 下载
    ModelUploadSession 导入
    基础错误提示和语言键

第三阶段：资源站 P1
    GitHub 根地址/tree URL
    默认资源站
    中文文件名安全 modelId

第四阶段：性能 P1/P2
    GPU 状态缓存
    AlphaProfile
    Lazy Loading
    LRU 卸载

第五阶段：NeoForge 独立迁移
    另起项目，从空 NeoForge 项目开始
```

### 如果只能先选一本计划书

先执行：

```text
Fox-Model-Loader 性能与多模型载入内存优化方案
```

原因：

```text
它是另外两个计划的稳定性前置条件。
资源站会增加模型来源和模型数量，性能问题会被放大。
NeoForge 迁移会复制当前问题，如果不先稳定 Fabric 版，迁移后会同时维护两套问题。
```

### 如果目标是尽快上线功能

折中顺序：

```text
1. 性能 P0
2. 资源站 P0 + P1
3. 性能 P1/P2
4. NeoForge 迁移
```

### 如果目标是尽快发布 NeoForge 版

可以调整为：

```text
1. 性能 P0
2. NeoForge P0/P1
3. NeoForge P2/P3
4. 资源站
```

但这会推迟玩家可见的新功能，并增加迁移期间的维护成本。
